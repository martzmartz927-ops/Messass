# Nexo — мессенджер с реальным бэкендом

Полноценное веб-приложение мессенджера: Node.js (Express) + SQLite + WebSocket для реального времени, файлы хранятся на диске сервера.

## Возможности

- Регистрация и вход (логин/пароль, хеш bcrypt, JWT-токены)
- Реальные чаты 1-на-1 с хранением истории в SQLite
- Сообщения приходят собеседнику мгновенно через WebSocket
- Индикатор "печатает…", статус "в сети" / "был(а) недавно"
- Отправка фото, файлов, стикеров (эмодзи) и своих GIF/стикеров (загружаются один раз, хранятся в личной библиотеке пользователя)
- Загрузка аватара профиля
- Поиск пользователей по @username, добавление в контакты
- Полностью рабочие настройки: аккаунт и смена пароля, уведомления, оформление (тема, акцентный цвет, размер шрифта, Enter для отправки), приватность (кто видит статус/фото), список активных сессий с возможностью завершения, автозагрузка медиа, язык, удаление аккаунта
- Никакого номера телефона — не используется нигде в приложении
- Название "Nexo" (латиница)

## Стек

- **Бэкенд:** Node.js + Express
- **База данных:** SQLite (better-sqlite3), файл создаётся автоматически
- **Реальное время:** WebSocket (библиотека `ws`)
- **Файлы:** хранятся на диске сервера в `server/uploads/{avatars,media,stickers}`, раздаются статически через `/uploads/...`
- **Аутентификация:** bcrypt + JWT (токен хранится в localStorage на клиенте)
- **Фронтенд:** ванильный HTML/CSS/JS (без сборки), один файл `public/app.js`

## Установка и запуск

Требуется Node.js 18+.

```bash
cd nexo
npm install
cp .env.example .env
# при желании отредактируйте .env (особенно JWT_SECRET для продакшена)
npm start
```

Сервер поднимется на `http://localhost:3000` (порт задаётся в `.env`).
Откройте этот адрес в браузере — это и есть само приложение (бэкенд отдаёт фронтенд статикой).

```
    ├── db/
    │   └── init.js        # схема SQLite и подключение
    ├── routes/
    │   ├── auth.js         # регистрация, вход, сессии
    │   ├── users.js        # профиль, аватар, поиск, контакты
    │   ├── chats.js        # чаты, сообщения, медиа, библиотека стикеров/GIF

        ├── media/
        └── stickers/
```

## Как это работает

1. При регистрации создаётся пользователь (пароль хешируется bcrypt), сразу создаётся строка настроек по умолчанию и выдаётся JWT-токен.
2. Токен кладётся в `localStorage` и подставляется в заголовок `Authorization: Bearer <token>` для всех запросов к `/api/...`, а также передаётся как query-параметр при подключении к WebSocket (`/ws?token=...`).
3. При открытии диалога клиент подгружает историю сообщений через `GET /api/chats/:id/messages`.
4. При отправке сообщения — `POST /api/chats/:id/messages` (текст/стикер) или `POST /api/chats/:id/media` (файл, `multipart/form-data`). Сервер сохраняет сообщение в SQLite и рассылает его всем участникам чата через WebSocket, поэтому оно появляется у собеседника мгновенно, без обновления страницы.
5. Файлы (аватарки, фото, GIF, стикеры) сохраняются на диск сервера в соответствующие подпапки `server/uploads/`, а в базе хранится только относительный путь. Статические файлы раздаются через `express.static` по пути `/uploads/...`.
6. Настройки хранятся в отдельной таблице `settings` (одна запись на пользователя) и полностью управляются через `GET/PATCH /api/settings`.

## Продакшен-заметки

- Обязательно замените `JWT_SECRET` в `.env` на длинную случайную строку.
- Для реального продакшена рекомендуется добавить HTTPS (через reverse-proxy, например nginx) — WebSocket тогда будет работать через `wss://`.
- SQLite отлично подходит для небольших/средних нагрузок; для масштабирования на несколько серверов потребуется вынести WebSocket-стейт в Redis/PubSub и перейти на PostgreSQL.
